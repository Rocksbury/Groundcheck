1. Purpose
What GROUNDCHECK is, what it tracks, and who it's for.
▸

GROUNDCHECK is a single-page readiness dashboard for 429 Squadron's AMSE (Aerospace Ground Support Equipment) and personnel posture. It consolidates data from squadron spreadsheets into one live board showing what is serviceable, what is short, and what is coming due — across three offices: SERC (ground support equipment & vehicles), AFSO (fluids & sealants), and ETO (training & qualifications).

Problem It Solves

Readiness data normally lives across several spreadsheets, each with its own layout, that someone has to read, cross-reference, and re-summarize for a brief or handover. GROUNDCHECK reads those same spreadsheets directly, applies the squadron's minimum-holding rules automatically, and renders the result as a single scored overview — with shortfalls, cal/inspection due items, and deployed equipment flagged at a glance.

Intended Users
RoleTypical Use
SERC Tool Crib / Equipment CustodiansDaily review of AMSE equipment, vehicle, ATGL, and palletized seat serviceability; deploy readiness checks.
AFSO SupplyFluids and sealant stock levels against minimum holdings, including expiry tracking on sealants.
ETO / Training OfficeTrade strength, qualification levels (C-Release, Level A, POM, Apprentice), and Tow Driver / Tow IC authorisation manning.
Supervisors / HandoverQuick readiness brief, printed/saved/emailed reports, and trend history across reporting cycles.
Offline Operation

GROUNDCHECK is a single self-contained HTML file. All parsing, scoring, charting, and report generation happen entirely in the browser — nothing is uploaded, and no server or network connection is required to operate it. The two third-party libraries it uses (SheetJS and Chart.js) are loaded from a CDN at startup; if they are already cached or the file has been adapted to bundle them locally, the tool runs fully offline. Theme preference is the only thing saved automatically, via the browser's local storage. All readiness history (snapshots) is stored in the user's own spreadsheets, not inside this file.

2. Daily Operation
Day-to-day use: loading data, reading the board, searching, and reporting.
▸
Loading Spreadsheet Data
1
What it does

Click ↑ Load Spreadsheet in the toolbar, or drag and drop a workbook anywhere on the page, to load one or more .xlsx/.xlsm files.

2
Why it is used

Each office maintains its own spreadsheet. GROUNDCHECK reads sheet names within each workbook to automatically classify and route the data to SERC, AFSO, or ETO — one, two, or all three can be loaded in the same session.

3
Expected result

The matching source badge (SERC / AFSO / ETO) in the toolbar lights up, the relevant board sections populate with tiles, and the banner at the top updates with that office's readiness numbers. If a sheet can't be classified or is missing expected data, the Data Warning banner appears above the toolbar — see Section 6, Error Reference.

Selecting a View Section & Reading the Banner
1
What it does

The View Section dropdown (top right of the banner) switches the summary banner between SERC, AFSO, and ETO.

2
Why it is used

Each office has a different scoring model (see Section 4 — Interface Reference, Dashboard Banner). The banner can only show one office's numbers at a time; the board below it shows tiles for all loaded offices simultaneously.

3
Expected result

The banner title, gauge, and the seven summary cells (Tracked Items, Score, Above Min, At Min, Below Min, Unserviceable, Cal/Insp) update to reflect the selected office. If that office has no data loaded, a "no data loaded yet" notice appears instead.

Searching & Filtering
1
What it does

The Search box filters tiles by name across all loaded sections as you type. The Filter buttons (ALL, BELOW MIN, AT MIN, U/S, CAL/INSP, DEPLOYED) restrict the board to tiles matching that status.

2
Why it is used

To quickly answer questions like "what's unserviceable right now" or "where is the RTV sealant" without scrolling the whole board.

3
Expected result

Matching tiles remain visible; everything else is hidden, and a match count appears next to the search box. If nothing matches, a "No tiles match your search or filter" message appears with a Clear all button. Press / to jump to the search box and Esc to clear it.

Reviewing Readiness Information (Tiles & Detail)
1
What it does

Each tile shows a status badge and colour, plus chips summarising counts (Serviceable, U/S, Cal/Insp, Deployed). Hovering a tile that has U/S, Cal/Insp, or Deployed items shows a popover with callsigns, locations, and notes. Some tiles (ATGLs, Palletized Seats) are clickable and open a detail window with full per-unit information.

2
Why it is used

The tile colour and chips give an at-a-glance status; the popover/detail window gives the specific callsign, location, or inspection due date needed to action it.

3
Expected result

A tile with no U/S, Cal/Insp, or Deployed items shows "No detail available" on hover — this is expected, not an error. A heavy horizontal line on the board marks the start of each office's section. Use ▲ Collapse All (key C) or keys 1/2/3 to focus on one office's sections, and 0 to expand everything again.

Running a "Can I Deploy?" Check
1
What it does

Click ✈ Can I Deploy? in the toolbar to compare current serviceable equipment counts and qualified personnel against fixed deployment minimums.

2
Why it is used

To get an immediate GO / NO-GO answer for a tasking, with the specific shortfalls identified, instead of manually checking each requirement against the board.

3
Expected result

A modal opens showing an overall GO — DEPLOYABLE or NO-GO — NOT DEPLOYABLE verdict, with a row per requirement (equipment item or trade qualification) showing have/need and MET/SHORT. The corresponding tiles on the board are also highlighted. Both the SERC and ETO spreadsheets must be loaded first — see Section 6, Error Reference, if you're prompted to load data.

Logging a Snapshot (for Trends)

Each loaded office's section has its own ↓ Log Snapshot button next to its section heading. It copies a row of that office's current numbers to the clipboard, dated to today, for pasting into a GC Log tab in that office's spreadsheet. The workflow is the same for SERC, AFSO, and ETO — only the columns differ.

1
One-time setup

Create a tab named exactly GC Log (case-sensitive) in that office's spreadsheet.

2
One-time header row

With the spreadsheet loaded, click Log Snapshot — a header row is copied to the clipboard. Paste it into cell A1 of the GC Log tab. Do this once only.

3
Each cycle

Click Log Snapshot again and paste the clipboard into the next empty row below the last entry. Do this on the same schedule each cycle (e.g. weekly).

4
Expected result

After reloading the spreadsheet and switching to the Trends tab, the SECTION (SERC/AFSO/ETO) and RANGE (6/12/24/all snapshots) selectors populate history charts from the GC Log rows.

What each office's snapshot contains

SERC — serviceable/U-S/cal/deployed counts per equipment item against its minimum, plus the data needed for the Chronic Issues summary (items below minimum in 4 or more logged snapshots). AFSO — quantity and status per fluid category and sealant against its minimum. ETO — "A-qualified"/total per trade, plus Tow Driver and Tow IC qualified counts against their minimums.

Renaming breaks history

Trend charts only plot items, fleets, or trades that exist in both the GC Log and the currently loaded spreadsheet. Renaming a tile, fleet, or trade in the live data will not retroactively rename it in the log — the old name's history will stop appearing.

Viewing Trends & Charts
1
What it does

The Trends tab plots logged snapshots as line charts (Chart.js), one set per office, plus a Chronic Issues list for SERC.

2
Why it is used

To see whether a shortfall is a one-off or a recurring pattern across reporting cycles, and to identify items that are persistently under minimum.

3
Expected result

Selecting SECTION and RANGE redraws the charts for that office and period. If a "No Trend Data" message appears instead, the GC Log tab is missing, empty, or has fewer than two data rows — see Section 5, Troubleshooting.

Generating & Printing Reports
1
Print Report

What it does: Click ⎙ Print (or press P). A modal lists SERC/AFSO/ETO, with only sections that have data loaded selectable. Why: produces a formatted, paper-friendly readiness summary. Expected result: confirming opens the browser's print dialog with an A4 landscape layout containing only the ticked sections.

2
Save Report

What it does: Click ⇩ Save Report. Why: produces a single downloadable HTML file ("429 Readiness Report <date>.html") containing a frozen copy of the current board, with all loaded data embedded. Expected result: opening the saved file in a browser shows the board exactly as it was at save time, with a "SAVED REPORT" banner across the top, and is itself a valid GROUNDCHECK file (it can be opened the same way as the main tool).

3
Email

What it does: Click ✉ Email. Why: generates the same file as Save Report and opens a modal with a pre-filled subject and body summarising each loaded office's banner stats. Expected result: the report file downloads automatically; use Copy Text to copy the summary or Open Email Client to launch a mailto: draft, then attach the just-downloaded report file manually — browsers cannot attach files to a mailto link automatically.

5. Troubleshooting
Common issues and how to resolve them.
▸
All SERC tiles show 0, or a Data Warning banner appears

Symptoms: AMSE or vehicle tiles all read 0 with the Data Warning banner showing. Likely cause: the workbook has no tab named exactly Critical AMSE Min Max List (for AMSE) or Vehicle List (for vehicles), or the tab returned no readable rows. Action: rename the tab to match exactly, or check that data starts at the expected row (see Section 7, Advanced Administration).

A specific SERC tile always shows 0

Symptoms: one tile never picks up rows even though the item exists in the spreadsheet. Likely cause: a keyword mismatch between the tile's kw list and the item name in column D of the spreadsheet (NSN codes are stripped automatically). Action: copy the exact item name text from column D and update the matching kw entry in ARULES/VRULES, in ALL CAPS — see Section 9, Code Modification Guide.

AFSO or ETO section doesn't appear on the board

Symptoms: a loaded workbook doesn't populate the AFSO or ETO board sections. Likely cause: none of the sheet names matched the auto-detection keywords for that office. Action: rename the sheet tab to include one of the recognised keywords (see Section 7, Advanced Administration — Auto-Detection Keywords).

"View Section" dropdown shows other offices greyed out

Symptoms: only one office is selectable in the View Section dropdown. Likely cause: the other offices' spreadsheets haven't been loaded yet in this session. Action: load the relevant spreadsheet(s) — the option becomes selectable once data is present.

Trends page shows "No Trend Data"

Symptoms: charts don't render for an office on the Trends tab. Likely cause: that office's spreadsheet has no GC Log tab, or it has fewer than two data rows. The tab name is case-sensitive. Action: follow the Logging a Snapshot steps in Section 2 to create the tab and log at least two snapshots.

An item, fleet, or trade is missing from a trend chart

Symptoms: one line/series doesn't appear even though other history is plotted. Likely cause: that item's name in the live spreadsheet no longer matches the name recorded in past GC Log rows (it was renamed). Action: keep tile/fleet/trade names consistent between the spreadsheet and the log; renaming breaks the historical link.

Chronic Issues list is empty even though an item is often short

Symptoms: an item that frequently shows below minimum doesn't appear under Chronic Issues on the SERC Trends view. Likely cause: by design, an item only appears once it has been below minimum in 4 or more logged snapshots (CHRONIC_MIN_FAILS). Action: continue logging snapshots each cycle; the threshold can be changed by an administrator (Section 7).

Hover popover shows "No detail available"

Symptoms: hovering a tile shows no useful detail. Likely cause: this is expected — popovers only have content for tiles with U/S, Cal/Insp, or Deployed items.

Print report is blank or missing a section

Symptoms: the print preview is empty, or one office's data doesn't appear. Likely cause: no spreadsheet loaded, or that office's section wasn't ticked in the print modal (only loaded sections are selectable). Action: load the spreadsheet(s) first, tick the desired sections, and confirm. If the print dialog itself doesn't open, ensure pop-ups aren't blocked — Chrome-based browsers are recommended.

Spreadsheet won't load / "Could not read" error

Symptoms: a notification reads "Could not read <filename>" with an error detail. Likely cause: the file isn't a valid .xlsx/.xlsm workbook, or it's corrupted/password-protected. Action: confirm the file opens normally in a spreadsheet application and is saved in a supported format, then reload.

Browser compatibility

Symptoms: layout, printing, or clipboard copy behaves inconsistently. Likely cause: older or non-Chromium browsers may differ in print-dialog and clipboard behaviour. Action: use a current Chromium-based browser (Chrome/Edge) for the most consistent results.

Theme resets between sessions

Theme is saved to the browser's local storage. If it resets every time, check that the browser isn't clearing site storage on exit, and that the file is being opened from the same path each time.

3. Shared Rendering Building Blocks
A
mkTileEl(opts)

The shared tile-card builder. Every equipment/fluid/sealant/training/ATGL/pallet tile is built from this one function, giving the consistent header / status badge / chip row / footer note layout.

B
mkchip(type, label)

Returns a small status "chip" (coloured dot + label) — types are svc (green), uns (red), cal (amber), dep (blue). Visual style lives entirely in CSS (.chip, .chip.svc, etc.).

C
_tileData (global array)

The single source of truth for search/filter. Every tile builder pushes a record here (with its DOM element and classification flags). applyViewState() reads it on every search/filter change and shows/hides/reorders tiles across all offices at once.

D
summarizeGrid(gridId, sumId)

Counts .tile elements in a grid and writes a one-line summary into the section header for the collapsed-section preview text.

E
showDetailModal(title, items, kind)

Generic detail modal used by clickable tiles (e.g. ATGLs, Palletized Seats) to show a per-unit breakdown.

4. Banner / Readiness Score

_bannerStats = {serc, afso, eto} holds each office's summary numbers as an array of {label, value, cls, score?} cells. SERC's stats are computed inline inside buildBoard() (folding in buildAtglSection()/buildPalletsSection() via _zeroStats()-shaped accumulators); AFSO and ETO have dedicated computeAfsoBanner() / computeEtoBanner() functions. renderBanner() reads whichever office is selected in the View Section dropdown and writes cell values into the DOM, including the gauge via the CSS custom properties --pct and --gauge-color.

6. Snapshots & Trends (History)

Each office has its own snapshot function (takeSnap() for SERC, takeAfsoSnap(), takeEtoSnap()) which builds a row of current stats and copies it to the clipboard — there is no automatic persistence; history lives in the user's spreadsheet. On reload, parseLog() / parseAfsoLog() / parseEtoLog() read those rows back, and renderSercTrends() / renderAfsoTrends() / renderEtoTrends() draw the Chart.js line graphs on the Trends page. To add a new tracked stat, update: the snapshot-builder function, logHeader() (header row text), and the matching parse*Log() + render*Trends() functions.

7. Deploy Readiness Check

checkDeployable() requires SERC and ETO data to be loaded. It compares snap.counts (per-equipment serviceable counts) against DEPLOY_EQUIP, and levelCounts() output from _trainData against DEPLOY_PERSONNEL, then calls showDeployModal(go, rows) to render the verdict and per-requirement table, and flags matching tiles with .deploy-ok/.deploy-flag.

8. Printing & Reporting

openPrintModal() shows a checklist of available sections (only offices with data loaded are selectable); confirmPrint() calls buildSercPrintSections() / buildAfsoPrintSections() / buildEtoPrintSections() to generate print-only markup (.pt-section, .pt-table, etc.), then triggers window.print(). The @media print CSS block hides interactive UI and shows only the generated .pt-* elements. saveReport() clones the live document, strips runtime state, embeds the current data as JSON in a #gc-snapshot script tag, and downloads it as a self-contained HTML file; loadSnapshot() restores that state on boot if the tag is present. emailReport() calls saveReport() then opens a pre-filled summary in the email modal.

9. Theming & Visual Conventions
•
CSS custom properties

All colours are defined as variables on :root (dark theme) with overrides under body.light (light theme), toggled by toggleTheme() and saved to local storage. Key variables: --bg, --panel, --panel2, --border, --blue/--bluhi, --amber, --green, --red, --dep, --muted, --lbl, --text. New UI should reuse these so both themes stay correct automatically.

•
Status colour convention

Green = good/met. Amber = attention. Red = problem. Blue = information. Keep this mapping when adding new statuses.

•
Fonts

Arial Narrow (--cond, condensed) for headers/labels/UI text; Consolas (--mono) for data values, timestamps, and the readiness gauge. Use these variables, not literal font names.

•
Accordion pattern

Both Guide sections and Board section headers use the same collapse pattern: a clickable header toggles a class, and CSS sibling/descendant selectors hide the body. Reuse this for any new collapsible block.

9. Code Modification Guide
Where it's safe to make changes, and where to be careful.
▸
LOW RISK — Text, Styling & Layout

Wording on labels, headings, guide content, tooltip text; CSS colours, spacing, fonts (use the existing --cond/--mono variables); rearranging static layout that doesn't change element ids referenced by JS. Test by reloading and visually checking the affected page.

MEDIUM RISK — Reporting Logic, Banner Calculations & UI Behaviour

Adding/removing/renaming entries in ARULES/VRULES/ATGLRULES; changing min values, AUTH_MIN, CHRONIC_MIN_FAILS, or DEPLOY_EQUIP/DEPLOY_PERSONNEL; adding a new banner cell to _bannerStats; adding a new filter button or keyboard shortcut. These change scoring/output but follow existing patterns — test by reloading with a real spreadsheet for the affected office and checking the banner, tiles, and (if relevant) print/report output and the Deploy Readiness Check.

HIGH RISK — Data Parsing, Data Model & Core Processing

Changing parseEquipment(), parseLiquids(), parseTraining(), parseAtgl(), parsePallets(), or any hardcoded row/column offsets; changing the shape of _tileData, snap.counts, _trainData, or the GC Log row format; changing mkTileEl()/mkchip()/applyViewState(); changing Chart.js dataset construction in render*Trends(). These affect every downstream feature (board, search/filter, printing, snapshots, trends). Trace the full path before changing it: processFiles() → a parse*() → the matching build*Section() → mkTileEl()/mkchip() → push to _tileData → applyViewState(). Any new module should follow this same shape to get search/filter, printing, and theming for free. Always keep a backup (previous version file) before editing.

Worked Example — Adding/Editing a SERC Equipment Tile (Medium Risk)
// Each ARULES/VRULES entry looks like this:
{key: 'nitrogen_cart', // unique ID — no spaces
 tile: 'Nitrogen Service Cart', // label on the tile
 kw: ['NITROGEN SERVICE CART'], // keywords matched against spreadsheet (ALL CAPS)
 min: 2}, // minimum serviceable required
+
To add a tile

Copy an existing entry, paste at the end of the array (before the closing ]), and update the four fields. The key must be unique. Keywords must match exactly how the item name appears in the spreadsheet — ALL CAPS, NSN codes already stripped.

−
To remove a tile

Delete the entire entry block. Check that surrounding commas are correct — no trailing comma on the last entry.

~
To change a minimum

Find the rule by its tile name and update the min number. Save and reload.

No Automated Tests
Manual verification required after any JS edit

There is no test suite. After any script change, at minimum: (1) extract each <script> block and pass it to new Function() in Node to catch syntax errors, and (2) load a real sample spreadsheet for each affected office and confirm tiles, banner, search/filter, snapshots, trends, deploy check, and print/save/email all still work.
